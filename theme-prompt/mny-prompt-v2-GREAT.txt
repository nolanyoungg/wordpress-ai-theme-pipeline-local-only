HEADER + NAV SYSTEM (DESKTOP + MOBILE)
========================================

Build a premium, production-grade header/navigation system with a solid `nolan-menu` header-menu interaction. This header must be fully designed, fully implemented, and pixel-polished for both desktop and mobile. The static preview must mirror the WordPress theme behavior exactly.

A) Header Layout (Desktop)
- Sticky header that becomes a "scrolled" variant after the user scrolls down.
  - On top: slightly taller padding, more airy.
  - Scrolled: slightly tighter padding, solid warm background, crisp border/shadow.
  - Must not cause layout shift.
- Structure (left -> center -> right):
  1) Logo block (left)
     - Mark + wordmark.
     - Mark can be inline SVG.
     - Wordmark is typographic, editorial, and premium.
     - Clicking logo goes to / (home).
  2) Primary nav (center)
     - Desktop nav items must be exactly:
       - What We Do
       - Who We Are
       - Work
       - Resources
       - Contact
     - What We Do is a button trigger that opens the `nolan-menu` panel for services.
     - Who We Are is a button trigger that opens the `nolan-menu` panel for studio/about content.
     - Resources is a button trigger that opens the `nolan-menu` panel for resources.
     - Work is a direct link to /work/.
     - Contact is a direct link to /contact/ and should also be represented by the header CTA.
     - Keep the primary nav readable, balanced, and visually calm.
  3) CTA area (right)
     - Primary CTA button text: Contact Us.
     - Primary CTA URL: /contact/.
     - CTA must be visible on desktop.
     - CTA should use .btn .btn-header-cta and feel like the strongest header action without overpowering the nav.
     - CTA should also be available in the mobile drawer as a full-width button.

B) Header Functionality (Desktop)
- Open and close `nolan-menu` panels on click.
- Hover-open is allowed and should be implemented as the user hovers over the drop-down menu options only if it is robust and does not break mobile or keyboard behavior.
- Only one panel may be open at a time.
- Opening a different panel closes the current panel.
- Clicking the same trigger again closes that panel.
- Clicking outside the open panel closes it.
- Pressing Escape closes the open panel.
- When a panel is open, show a full-page backdrop.
- When a panel is open, lock body scroll.
- All trigger buttons must maintain accurate aria-expanded values.
- Trigger buttons must use aria-controls where appropriate.
- Panels must be hidden from assistive technology when closed (use hidden, aria-hidden, or equivalent state management correctly).
- Focus-visible states must be strong and obvious.
- No keyboard traps.
- No console errors.
- Panels must be SOLID and readable (warm white/ivory background, border, shadow, stable z-index). No transparency.
- Panel layout: left rail (list of category buttons) + right content area with multiple sections.
- Right content MUST update on rail hover AND rail keyboard focus (Tab navigation). Keyboard focus behavior is required, not optional. This is the signature interaction.

C) Required Desktop Data Attributes (Follow Exactly)
- button[data-menu-item="what-we-do"] controls div[data-menu-dropdown="what-we-do"]
- button[data-menu-item="who-we-are"] controls div[data-menu-dropdown="who-we-are"]
- button[data-menu-item="resources"] controls div[data-menu-dropdown="resources"]
- Trigger buttons should use aria-controls to reference the matching panel ID.
- Trigger buttons should update aria-expanded when panels open/close.

Inside each open header panel:
- Left rail buttons: button[data-rail-item="<key>"]
- Right sections: section[data-rail-content="<key>"]
- Only one matching right-side section visible at a time.
- Closed/inactive right-side sections should not expose hidden focusable controls to keyboard users.

- Each desktop header panel must be solid, readable, and premium.
- Use a warm white or ivory background.
- No transparent panels.
- No glassmorphism that hurts readability.

- Panel should sit below the header and feel connected to it.
- Panel layout must use a left rail and right content area for What We Do and Who We Are.
- Left rail contains category buttons.
- Right content area contains matching content sections.
- The right content must update on left rail hover and on left rail keyboard focus.
- Only one right-side content section should be visible at a time.

CSS/SCSS checklist:

- No horizontal overflow on mobile.
- No transparent mega panels.
- No unreadable overlays.

WORDPRESS + PREVIEW + GITHUB PAGES RULES
========================================

Build two matching deliverables:

Important distinction:
- WordPress runs PHP templates.

WordPress requirements:
- Classic theme templates (PHP), not a block theme.

- Proper enqueueing via functions.php.
